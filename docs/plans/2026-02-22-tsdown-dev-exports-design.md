# tsdown Dev Exports Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Enable editor go-to-definition to resolve to `.ts` source files across orval's 13 packages using tsdown's `devExports` feature.

**Architecture:** Change `exports: true` to `exports: { devExports: true }` in the shared tsdown base config. tsdown auto-generates each package.json with source exports for local dev and `publishConfig` with built output for publishing. Zero import path changes needed.

**Tech Stack:** tsdown 0.20.3, TypeScript (bundler resolution), Bun workspaces

---

## Context

- **Monorepo**: 13 packages in `packages/`, all using shared `packages/tsdown.base.ts`
- **Current behavior**: `exports: true` makes tsdown auto-generate `exports` pointing to `./dist/index.mjs`
- **Desired behavior**: `exports: { devExports: true }` makes tsdown auto-generate `exports` pointing to `./src/index.ts` with `publishConfig` overriding for publish
- **Why no .ts extensions needed**: Dev exports only change entry point resolution. Internal imports use `moduleResolution: "bundler"` which resolves extensionless imports fine.

## Problem Statement

PR #2544 attempted dev exports but required adding `.ts` extensions to ALL internal imports (3060-line diff, 100+ files). This is unnecessary — only the tsdown config needs to change.

## Alternatives Considered

| Approach                    | Changes             | Why not                                                        |
| --------------------------- | ------------------- | -------------------------------------------------------------- |
| `devExports: 'development'` | 2 files             | More complex, npm compat irrelevant for bun project            |
| TypeScript `paths`          | 1 file, 12+ entries | Maintenance burden, TS team discourages                        |
| Project References          | 12+ files           | Overkill, incremental tsc builds don't apply with tsdown       |
| PR #2544 (.ts extensions)   | 100+ files          | Unnecessary — bundler resolution handles extensionless imports |

### Task 1: Update tsdown base config

**Files:**

- Modify: `packages/tsdown.base.ts:12`

**Step 1: Change exports config**

Change line 12 from:

```ts
  exports: true,
```

to:

```ts
  exports: { devExports: true },
```

Full file should be:

```ts
import type { UserConfig } from 'tsdown';

export const baseConfig: UserConfig = {
  entry: ['src/index.ts'],
  target: 'node22.18',
  platform: 'node',
  format: 'esm',
  tsconfig: 'tsconfig.build.json',
  dts: {
    sourcemap: true,
  },
  exports: { devExports: true },
};
```

**Step 2: Commit**

```bash
git add packages/tsdown.base.ts
git commit -m "feat: enable tsdown devExports for editor DX"
```

## Risk Mitigation

- If tsdown doesn't generate the expected package.json structure, fall back to `devExports: 'development'` (Approach 2 from design doc) which uses conditional exports instead of publishConfig
- If the orval CLI multi-entry export is wrong, the `orval` package can use a custom tsdown config that overrides the exports behavior
- All changes are to build config and auto-generated fields only — no source code is touched, so rollback is trivial
